## Задание 3. Репликация
           
### Серверная часть

* В сервис `KVService` или `KVCluster` добавить поддержку хранения нескольких копий данных для каждого ключа.
* Можно реализовать существующий интерфейс `ReplicatedService`.
* Фактор репликации `n` (количество узлов-реплик) настраивается при запуске сервиса (через CLI-аргумент, ENV-переменную или конфиг-файл).
* В качестве узлов хранения допустимо использовать файлы на диске (по одному файлу на реплику/узел) или массив, где индекс -- это номер узла, а значение -- "узел" в виде хранилища в памяти.
                                                                                                        
### API

* Добавить query-параметр `ack` к эндпоинтам чтения/записи. Он определяет, какое число реплик должно успешно подтвердить операцию.
* Обеспечить полную обратную совместимость с предыдущей версией API (запросы без `ack` должны обрабатываться корректно, с дефолтным поведением, например `ack = 1`).

| Метод                                                              | Поведение                                                                                        |
|--------------------------------------------------------------------|--------------------------------------------------------------------------------------------------|
| `GET /v0/entity?id=ID&ack=X`                                       | Возвращать данные, если подтвердили ответ ≥ `ack` реплик. Если доступно меньше – возвращать 50x. |
| `PUT /v0/entity?id=ID&ack=X` <br/> `DELETE /v0/entity?id=ID&ack=X` | Ждать подтверждений от ≥ `ack` реплик. При недоступности нужного числа – возвращать 50x.          |
| Валидация                                                          | Если `ack > n`, немедленно возвращать 400 Bad Request.                                           |
  
### Интеграционный тест

* Значения по одному и тому же ключу всегда должны попадать на одни и те же реплики.

* Обеспечить graceful degradation: система должна корректно обрабатывать запросы при частичной недоступности реплик, соблюдая заданный уровень ack.

## Задание 5. Консенсус (выбор узла-лидера).

Цель &mdash; попрактиковаться в реализации одной из ключевых частей алгоритма распределенного консенсуса, а именно выбора
узла-лидера. Освоить принципы работы многопоточных распределенных систем, механизмы обработки отказов узлов,
мониторинга доступности и обеспечения строгой консистентности состояния кластера.

### Модель распределенной системы

* Создать отдельное Java-приложение, представляющее модель распределенной системы из `N` узлов.
* Каждый узел системы реализуется как отдельный поток (`Thread`) с независимым локальным состоянием.
* У каждого узла должен быть заданный уникальный идентификатор (ID в виде целого числа).
* Каждый узел имеет доступ к остальным узлам кластера и знает их идентификаторы.
* Узлы пересылают между собой сообщения мгновенно (через прямой вызов методов).
* Для упорядоченного и потокобезопасного приема входящих сообщений у каждого узла должен быть организован `BlockingQueue`.

### Обработка отказов и восстановления узлов
* Во время работы каждый процесс-узел может с заданной вероятностью имитировать отказ («выход из строя») и спустя случайное или заданное время восстанавливаться.
* Поведение моделируется периодической генерацией случайного числа (например, «бросок кубика» раз в несколько секунд).
* Для удобства тестирования реализовать возможность принудительного включения/выключения узла по внешнему сигналу (через вызов метода/API).
* Отказавший узел перестает отвечать на запросы от других узлов и не отправляет ответные сообщения до момента восстановления.

### Логика выбора лидера
* В любой момент времени система должна стремиться к состоянию, когда существует один и только один активный узел-лидер. Ситуация с двумя лидерами (split-brain) недопустима.
* Лидером должен выбираться доступный узел с наибольшим ID.
* Выбор лидера запускается автоматически в следующих случаях:
  * При старте системы (инициализация кластера).
  * Если узел обнаружил, что лидер недоступен (например, очередной ping не получил ответ за заданное время таймаута).
  * Если узел восстановился после отказа.
* Реализовать корректную обработку таймаутов, предотвращение гонок данных (race conditions) и гарантию идемпотентности переходов состояний.

### Типы пересылаемых сообщений
* `PING` – периодически отправляется узлу-лидеру от ведомых узлов, чтобы определить, что лидер доступен.
* `ELECT` – рассылается узлом-инициатором узлам с бОльшими ID в начале проведения выборов.
* `ANSWER` – ответ на `PING` или `ELECT`.
* `VICTORY` – это сообщение выбранный узел-лидер (то есть тот узел, который не получил ни одного `ANSWER` на `ELECT` от старших узлов) отправляет всем остальным узлам, чтобы они зафиксировали у себя в памяти ID лидера.
* Любое сообщение может содержать ID узла-отправителя и любую другую метаинформацию. 

### Тестирование и отчёт

Провести ручное и/или автоматизированное тестирование работы алгоритма в различных сценариях отказоустойчивости.

Сценарии для проверки:
* Корректный старт: выбор лидера при инициализации кластера без отказов.
* Отказ лидера: текущий лидер падает, система детектирует недоступность и выбирает нового лидера (следующий по максимальному ID среди живых).
* Восстановление узла: ранее упавший узел возвращается в кластер. Проверить, что он корректно синхронизируется и не нарушает текущий консенсус.
* Частые отказы/восстановления: проверка стабильности алгоритма при высокой частоте смены состояния узлов.

### Бонусные задания (по желанию)
* Визуализация состояния кластера. Реализовать отображение топологии и статусов узлов (Leader / Follower / Down) в реальном времени (например, через консольный UI, Swing/JavaFX или вывод структурированных логов с цветовой разметкой).

## Задание 6. Введение в Apache Kafka

Цель - изучить как конфигурировать клиента, писать и читать из Kafka.

### Серверная часть

В задании необходимо реализовать механизм аудита действий над **entity** в `KVService`. Факт каждого обращения к ресурсу
должен попадать в сервис аудита. Сервис аудита должен выдавать полный список аудитных событий по запросу.

* Для выполнения задания использовать интерфейс `AuditableKVService`.
* Для хранения аудита использовать `AuditService`.
* Факт каждого запроса по ресурсу `/v0/entity` в `AuditableKVService` должен отправляться в Kafka топик `audit`.
* Что отправлять в `AuditEvent`: имя http метода, параметр id из запроса, время получения http запроса.
* `AuditService` читает данные из Kafka топика `audit` и сохраняет их в БД или в файле. Рестарт экземпляра `AuditService` не должен приводить
к повторному вычитыванию уже полученных событий.
* Метод `AuditableKVService.setAsync(boolean)` должен включать асинхронный (true) режим отправки сообщения в Kafka или синхронный (false). 
* Метод `AuditService.stop()` должен останавливать сервис и отключать consumer от Kafka.
* Метод `AuditService.start(consumerGroupId)` должен запускать экземпляр сервиса с подключением к Kafka с указанной consumerGroupId.

### Тесты 

Основные сценарии можно проверить, запустив интеграционный тест `KafkaTest`.

### Бонусное задание

С помощью утилиты wrk провести нагрузочное тестирование в синхронном и асинхронном режиме отправки аудита:
* На графике должно быть две линии - синхронный режим и асинхронный режим
* График должен отображать время выполнения запросов (p50, p95, p99, RPS)
